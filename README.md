# 旅遊總管

本專案旨在打造一個**輕量化、去中心化且兼具高效能的多人即時協作旅遊規劃系統**。

###  開發初衷與核心理念
在現有的旅遊規劃工具中，使用者常面臨兩大痛點：一是平台的架構過於繁複，缺乏隱私分流；二是跨裝置協作時，資料常受限於特定帳號系統。

本系統的開發初衷在於**回歸「純粹的行程分享與極簡操作」**，打破傳統 App 需要註冊、集中式管理的繁瑣流程，讓使用者能如同分享房間鑰匙般，透過單一協作連結即刻啟動跨裝置、跨平台的即時同步。

### 核心技術挑戰與突破

身為前端與行動端的網頁應用，本專案在技術實作上達成了以下關鍵突破：

* **自研行動端觸控拖曳演算法**：針對 HTML5 原生 Drag and Drop API 於行動端瀏覽器（iOS/Android Safari/Chrome）失效的限制，本專案不依賴龐大的第三方套件，純手工透過 React `useRef` 與觸控事件（`Touch Events`），實作了一套基於 `document.elementFromPoint` 的實時座標動態比對與碰撞偵測機制，大幅提升行動端的交互流暢度（UX）。
* **非集中式的數據持久化與分流**：系統設計了智慧分流機制，利用本地 LocalStorage 進行使用者自建（My Trips）與外部接收（Shared Trips）的清單管理，既維持了使用者個體的「去中心化」私密空間，又兼顧了協作群組間的雙向 Firestore 實時即時數據串流。
* **狀態競爭防禦**：針對多人實時協作可能引發的異步數據衝突，優化了生命週期監聽器的重啟與校驗鎖，徹底杜絕 Day ID 異步加載時的數據丟失，確保高併發協作下的資料一致性。

這不僅是一個實用的旅遊 App，更是一個針對行動網頁原生限制、即時同步與分散式快取設計的完整技術解決方案。

##  核心功能與亮點

### 1. 多人即時協作與智慧分流
* **單一連結邀請**：透過產生帶有群組特徵（`groupId`）的協作網址，朋友點擊即可開啟相同頁面。
* **智慧列表分流**：
  * 使用者自行點擊「建立新旅程」產生的項目，將永久歸類於 **「 我建立的行程」**。
  * 點開朋友分享的連結，系統會進行解構判定，自動將其歸類於首頁的 **「朋友分享的行程」**。

### 2. 📱 行動端觸控拖曳排序 
* 針對 iOS/Android 瀏覽器不支援 `draggable` 屬性的原生缺陷，自行透過 React `useRef` 與 **Touch Events (`onTouchStart`, `onTouchMove`, `onTouchEnd`)** 實作了一套**碰撞偵測演算法**。
* 透過 `document.elementFromPoint(x, y)` 動態比對網頁座標，在手指滑動時即時進行微型 UI 陣列排序，並在手指放開後**批次寫入（Firestore WriteBatch）** 雲端，兼顧效能與操作手感。
* 容器設定 `touch-action: none` 防止系統預設的下拉重整攔截拖曳行為。

### 3. 💾 強效資料同步與防丟失機制 
* **狀態競爭處理**：在多人同時對一趟旅程進行天數（Day ID）與行程切換時，`useEffect` 會強制解構並清除舊的監聽器（Cleanup Subscription），即時與雲端重新建立精確連線，防止資料寫入錯誤路徑。
* **防呆安全鎖**：新增行程前會自動校驗 `tripId` 的有效性，避免因為網路延遲將數據存入「虛無」中。

* **雙向名稱同步**：不論在行程內或首頁大廳更名，變更將同時推播至雲端及本地 `LocalStorage`快取，確保所有協作人員介面一致。

### 4.  精確的大廳與內部編輯模式
* 首頁旅程名稱採用**非即時觸發的獨立編輯按鈕「📝」設計**，避免滑動時誤觸。
* 行程內部（卡片備註）支援 **網址自動偵測**，會透過 Regex 自動篩選出連結並轉化為帶有 `target="_blank"` 的可點擊超連結，並一鍵對接 Google Maps API 提供實時地圖查詢。

##  技術棧 

* **Framework**: Next.js 14+ (React 18+, TypeScript)
* **Styling**: Tailwind CSS / Inline CSS (Flexbox & Grid 響應式佈局)
* **Database**: Firebase Firestore (Real-time `onSnapshot` 異步數據串流)
* **State Management**: React Hooks (`useState`, `useEffect`, `useRef`)
* **Storage**: LocalStorage API (快取與自建/分享清單分流持久化)

## 系統架構與資料流 
### Firestore 資料庫設計
* **`groups/{groupId}`**: 儲存旅程名稱及天數陣列（如：`[{id: "day_1", label: "Day 1"}]`）。
* **`trips/{dayId}/plans/{planId}`**: 儲存特定天數下的具體行程（地點、備註、`order` 排序欄位）。

## 開發挑戰與除錯歷程 

在開發本專案的過程中，團隊面臨了從前端交互、即時同步到後端架構等多個維度的技術挑戰，並透過敏捷開發與持續整合測試逐一克服：
1. 跨端異步數據同步與狀態競爭 
    面臨困難：本專案的初衷是打造如 Apple 備忘錄般的多人高效協作體驗。但在跨裝置（手機對手機）即時測試時，發現當 A、B 使用者同時變更天數標籤（Day ID）或快速新增行程時，由於 Firebase Firestore 異步載入的時間差，導致數據偶爾會寫入錯誤的虛擬路徑，造成「行程短暫消失」的狀態競爭現象。

    除錯與解決方案：與 AI 協同檢視生命週期與 React Hook 依賴。最終在 handleAddPlan 與 useEffect 中導入了防禦性狀態校驗鎖。在切換天數時，強制清除舊有的異步監聽器並重新綁定合法的 tripId，確保多端高併發修改時的資料一致性。

2. 行動端觸控流暢度與碰撞偵測

    面臨困難：為了提升使用者調整行程時的流暢度，團隊不希望採用傳統「置頂/置底」的定序按鈕，而是追求「隨意滑動、即時內插交換」的直覺體驗。然而 HTML5 原生的拖曳 API 在行動端瀏覽器（iOS/Android）原生失效，導致手機端操作嚴重卡頓。

    除錯與解決方案：團隊引進了敏捷的使用者回饋機制，根據朋友即時測試的流暢度反饋，捨棄了龐大的第三方拖曳庫，純手工利用 TouchEvent (Start/Move/End) 捕捉觸控軌跡。透過 document.elementFromPoint(x, y) 動態計算手指當前網頁座標，實作了一套即時碰撞偵測演算法。並在容器層強制宣告 touch-action: none 防止網頁原生捲動攔截，達成了媲美原生 App 的滑動排序體驗。

3. 跨平台 API 整合與第三方路由跳轉

    面臨困難：為提升行程規劃的實用性，系統需整合外部資訊。在串接 Google Maps API 做地理位置導航時，需要確保動態編碼的正確性，避免特殊字元導致跳轉失敗；同時，為了豐富社群擴充性，也嘗試透過 URL Scheme 導向 Threads 等社群平台的熱門推薦貼文（如：美食、購物建議），在跨平台的 Webview 跳轉與狀態保持上做了多次調校。

4. 後端架構限制與雲端成本權衡 

    後續瓶頸：本系統目前採用 Firebase Firestore 的免費層級進行開發。由於免費額度存在每日讀寫次數以及單一文件容量限制，在多人頻繁進行即時協作測試後，偶爾會觸及後端配額上限，導致歷史紀錄無法成功載入、或新行程寫入失敗。

    未來改進方向：此問題屬於雲端服務供應商的計費架構限制。未來若投入商用，系統架構預計優化為混合式架構：在前端引入 Redis 等快取機制，或引入更激進的本地快取機制（如：IndexedDB / 強化 LocalStorage 離線同步），在本地完成批次聚合後再寫入雲端，以大幅降低對後端資料庫的直接請求頻率。

📝 專案開發心得 

透過本專案的迭代開發，我深刻體會到「軟體測試」不能僅停留在開發者的局部環境，必須在設計初期就構想多種未來可能發生的邊界條件。透過邀請真實使用者進行多端並行測試、收集即時流暢度回饋，並反覆與 AI 進行架構層面的代碼重構，才得以符合使用者預期。
